Giải thích luồng chạy quan trọng:

  1. Xác thực tập trung ở API Gateway: Khi Client gửi HTTP request, API Gateway chặn lại tại JwtAuthGuard. Nó giải mã token JWT bằng Secret Key chung. Nếu hợp lệ, thông tin User (gồm ID và Role) sẽ được đính vào đối tượng request.
  2. Truyền danh tính qua Header: Gateway chuyển tiếp request đến dịch vụ con thông qua http-proxy-middleware. Đồng thời, nó thêm các header tùy biến:
    • x-user-id: Để dịch vụ con biết ai đang gọi (ví dụ: tạo thú cưng lấy ID làm owner_id).
    • x-user-role: Để dịch vụ con phân quyền tài nguyên (nhờ RolesGuard ở nội bộ dịch vụ con đó).
  3. Bảo mật các dịch vụ con: Các dịch vụ con (User, Pet, Vaccine,...) chạy trong mạng ảo Docker và không expose port ra bên ngoài (chỉ expose trên môi trường dev, môi trường chạy thật sẽ chặn port bên ngoài, chỉ cho phép kết nối từ container gateway).

Thiết Kế Cơ Sở Dữ Liệu Phân Tán

Chi tiết cấu hình Database per Service giúp tối đa hiệu suất và tính cô lập nghiệp vụ.

Danh sách 7 Database khởi tạo trong PostgreSQL:

Tên Database Dịch vụ sở hữu Bảng & Quan hệ chính
pet_care_auth auth-service Bảng users (id, email, password_hash, role, full_name, created_at)
pet_care_pets pet-service Bảng pets & breeds (pets liên kết breed_id, owner_id)
pet_care_vaccine vaccine-service Bảng pet_vaccination & vaccine_category
pet_care_appointment appointment-service Bảng appointments (user_id, pet_id, vet_id, date, status)
pet_care_medical_records medical-record-service Bảng medical_records (hồ sơ bệnh án)
pet_care_nutrition nutrition-service Bảng nutrition_plans & foods
pet_care_review review-service Bảng reviews (id, user_id, vet_id, rating, comment)

Lưu ý về liên kết cơ sở dữ liệu:

Trong kiến trúc này, không có bất kỳ khóa ngoại vật lý (Foreign Key constraints) nào liên kết chéo giữa các database khác nhau. Ví dụ: Cột owner_id trong database pet_care_pets chỉ lưu mã ID dạng số nguyên. Tầng ứng dụng (NestJS hoặc Frontend React) chịu trách nhiệm thực thi các truy vấn gián tiếp để lấy thông tin thay vì SQL JOIN chéo máy chủ.

Bộ Câu Hỏi & Trả Lời Phản Biện Chi Tiết

Danh sách toàn bộ các câu hỏi kèm câu trả lời đã viết sẵn chi tiết cho bạn ôn tập.

Trả lời: Hệ thống được thiết kế theo kiến trúc Microservices với các thành phần chính:

  • Client: Gồm React Web (Vite/TS) cho Quản trị viên/Bác sĩ và React Native (Expo) cho Khách hàng.
  • API Gateway (NestJS): Nhận tất cả request từ client, xử lý xác thực (JWT) và định tuyến (proxying) xuống các dịch vụ con.
  • Downstream Services (Dịch vụ con): Auth-service, User-service, Pet-service, Vaccine-service, Appointment-service, Medical-record, Nutrition, Review. Các dịch vụ này chạy độc lập.
  • Database: Mỗi dịch vụ sở hữu cơ sở dữ liệu PostgreSQL của riêng mình để đảm bảo tính độc lập và tự trị (Database-per-service).

Mẹo: Nhấn mạnh việc chuyển tiếp thông tin user qua header x-user-id từ Gateway.

Trả lời:

  • Lý do chọn: Dự án hướng tới một hệ sinh thái chăm sóc sức khỏe mở rộng. Việc tách biệt giúp hệ thống dễ dàng mở rộng độc lập từng dịch vụ (ví dụ: dịch vụ đặt lịch hẹn cao điểm có thể nâng cấp CPU/RAM riêng biệt mà không ảnh hưởng dịch vụ dinh dưỡng).
  • Ưu điểm: Cô lập lỗi (dịch vụ dinh dưỡng hỏng thì đặt lịch tiêm phòng vẫn chạy), tự trị công nghệ (có thể viết một dịch vụ bằng Go/NodeJS/Python tùy ý), dễ phân công nhân sự phát triển song song.
  • Nhược điểm: Hệ thống phức tạp trong việc quản lý, deploy (cần Docker), khó khăn hơn khi xử lý giao dịch phân tán (Distributed Transactions) và join dữ liệu chéo.

Trả lời: API Gateway trong dự án đóng vai trò:

  • Cổng truy cập duy nhất (Single Entry Point): Cung cấp một địa chỉ duy nhất cho Client gọi tới (Port 3000), ẩn đi toàn bộ cấu trúc cổng phức tạp của các dịch vụ con bên trong.
  • Xác thực & Phân quyền tập trung: Kiểm tra chữ ký số token JWT gửi từ Client.
  • Định tuyến động (Dynamic Routing): Định tuyến request tới đúng dịch vụ đích dựa trên Path URL prefix.

Em tự xây dựng API Gateway bằng NestJS và sử dụng thư viện trung gian http-proxy-middleware để tự động proxy request xuống các dịch vụ đích một cách mềm dẻo.

Trả lời: Hiện tại trong dự án, các dịch vụ giao tiếp chủ yếu theo cơ chế Đồng bộ (Synchronous) qua các HTTP REST request chuyển tiếp từ API Gateway. Gateway đóng vai trò làm trung gian luân chuyển thông tin.

Định hướng nâng cấp trong tương lai là tích hợp thêm cơ chế Bất đồng bộ (Asynchronous) thông qua Message Broker như RabbitMQ hoặc Kafka để xử lý các tác vụ như gửi thông báo (Notification) hoặc đồng bộ dữ liệu người dùng khi có sự thay đổi trạng thái ở Auth Service.

Trả lời: Hệ thống sử dụng token JWT (JSON Web Token):

  • Xác thực: Khi đăng nhập thành công ở Auth-service, Client nhận về một access_token và lưu vào LocalStorage (với Web) hoặc Secure Store (với Mobile). Khi gọi API tiếp theo, client đính token này vào header Authorization: Bearer <token>. API Gateway sẽ chặn lại và giải mã bằng JwtAuthGuard để nhận diện người dùng.
  • Phân quyền: Các API yêu cầu quyền hạn cụ thể (ví dụ: Vet hoặc Admin) sẽ được gắn decorator @Roles(Role.ADMIN, Role.VET). Dịch vụ con sử dụng RolesGuard để kiểm tra quyền hạn lưu trong header x-user-role do gateway đính kèm xuống, nếu hợp lệ mới thực thi tiếp logic.

Trả lời: Việc kiểm tra JWT tập trung tại API Gateway mang lại 2 lợi ích cốt lõi:

  • Tránh lặp lại code (DRY - Don't Repeat Yourself): Nếu không có Gateway, từng dịch vụ con (Pet, Vaccine, Appointment,...) đều phải viết logic giải mã JWT và giữ Secret Key để verify. Điều này gây dư thừa code và khó bảo trì khi thay đổi cơ chế xác thực.
  • Tối ưu hiệu năng: Các request không hợp lệ (hết hạn token, sai chữ ký) sẽ bị Gateway từ chối ngay lập tức, không đi sâu vào mạng nội bộ của các dịch vụ con, giúp tiết kiệm băng thông và tài nguyên CPU/Database hạ tầng.

Trả lời: Trên môi trường phát triển (Development), các cổng (port 3001, 3002, 3003...) được khai báo expose trong file docker-compose.yml để thuận tiện cho việc debug và test API lẻ.

Tuy nhiên, trên môi trường chạy thực tế (Production), chúng ta sẽ bỏ dòng expose ports này ra ngoài. Các container dịch vụ con chỉ giao tiếp với nhau trong Docker Network nội bộ (pet-network). Cổng duy nhất mở ra Internet là cổng 3000 của API Gateway. Hacker hoàn toàn không thể kết nối trực tiếp đến cổng của dịch vụ con vì các cổng đó không được ánh xạ (map) ra ngoài máy chủ vật lý.

Trả lời:

  • Lý do chọn: Đúng triết lý Microservices. Đảm bảo tính tự trị cao nhất. Schema database của dịch vụ này thay đổi không ảnh hưởng đến dịch vụ khác. Nếu database của Pet Service hỏng, dịch vụ Auth hay Dinh dưỡng vẫn hoạt động độc lập bình thường.
  • Thách thức lớn nhất: Tính nhất quán dữ liệu (Data Consistency) và truy vấn dữ liệu liên kết. Vì không thể dùng SQL JOIN chéo giữa các database ở máy chủ khác nhau, nên khi cần hiển thị dữ liệu tổng hợp, hệ thống phải thực hiện API Composition ở gateway/client, khiến số lượng HTTP request tăng lên.

Trả lời: Trong hệ thống này, em áp dụng phương pháp API Composition (Gộp dữ liệu ở tầng ứng dụng):

  1. Client gọi tới Gateway để lấy danh sách lịch hẹn. Lịch hẹn trả về có chứa các trường ID liên kết: pet_id, vet_id.
  2. Client hoặc API Gateway sẽ gửi tiếp các request song song xuống:
    • Pet Service để lấy thông tin chi tiết của thú cưng từ danh sách pet_id.
    • User Service để lấy thông tin tên, số điện thoại của bác sĩ thú y tương ứng với vet_id.
  3. Gộp dữ liệu đó lại trên Client để render giao diện hoàn chỉnh cho người dùng.

Trả lời: Guard trong NestJS là các class được đánh dấu bằng decorator @Injectable(), kế thừa interface CanActivate. Guard chịu trách nhiệm quyết định xem một request có được phép truy cập vào router handler hay không dựa trên các điều kiện xác định (như quyền hạn, token hợp lệ).

Dự án sử dụng:

  • JwtAuthGuard ở API Gateway: Kiểm tra token JWT hợp lệ từ Header.
  • RolesGuard ở các dịch vụ con: Đọc thông tin vai trò (x-user-role) và đối chiếu với yêu cầu của API (qua `@Roles(...)`) để cho phép hoặc chặn (trả về lỗi 403 Forbidden).

Trả lời: Dạ, đây là định hướng chuẩn bị hạ tầng phục vụ việc nâng cấp hệ thống (Scalability):

  • Hiện tại, hệ thống đã cài đặt sẵn container Redis (Port 6379) và khai báo liên kết phụ thuộc (depends_on) trong docker-compose ở các dịch vụ để chuẩn bị sẵn hạ tầng bộ nhớ đệm Cache.
  • Khi lượng truy cập tăng cao, em sẽ sử dụng Redis để lưu trữ tạm thời các dữ liệu ít biến động nhưng tần suất đọc lớn (như danh mục loại vaccine, danh sách bác sĩ thú y) để giảm tải truy vấn trực tiếp vào database PostgreSQL, tăng tốc độ phản hồi API dưới 10ms.
  • Đồng thời, Redis có thể dùng làm kho lưu trữ Session hoặc danh sách đen JWT Token (Blacklist Token) khi người dùng nhấn Logout.

Trả lời: Dự án sử dụng thư viện bcrypt với số vòng mã hóa (salt rounds) là 10 để băm mật khẩu trước khi lưu trữ vào bảng users ở database pet_care_auth.

Salt (Muối) là một chuỗi ngẫu nhiên được cộng thêm vào mật khẩu trước khi thực hiện quá trình băm (hash). Việc sử dụng Salt giúp tránh cuộc tấn công tra cứu bảng băm sẵn (Rainbow Table). Dù hai người dùng có mật khẩu giống hệt nhau (ví dụ: "123456"), chuỗi băm lưu dưới database của họ vẫn sẽ hoàn toàn khác nhau nhờ chuỗi muối ngẫu nhiên này.

Trả lời: Trong hệ thống, ở một số dịch vụ nghiệp vụ như vaccine-service, em tận dụng Lifecycle Hook OnApplicationBootstrap của NestJS:

  • Khi ứng dụng bắt đầu khởi chạy hoàn tất, phương thức onApplicationBootstrap() sẽ tự động kiểm tra xem bảng vaccine_category có dữ liệu hay chưa.
  • Nếu số lượng bản ghi bằng 0, nó sẽ đọc file mẫu JSON chứa dữ liệu tiêm chủng (vaccine.json) thông qua thư viện đọc file fs của NodeJS, chuyển đổi (parse) thành Object và lưu hàng loạt vào database PostgreSQL.

Trả lời: Cloudinary được sử dụng làm dịch vụ lưu trữ đám mây cho các tài nguyên hình ảnh (như ảnh đại diện của thú cưng, ảnh hồ sơ bệnh án). Khi người dùng tải ảnh lên, file sẽ được truyền qua controller (sử dụng FileInterceptor và Multer) gửi trực tiếp tới Cloudinary API. URL ảnh trả về từ Cloudinary sẽ được lưu vào PostgreSQL.

Lợi ích: Tránh lưu trữ file cục bộ trên các server container Microservices (vì container có tính chất tạm thời, dễ bị xóa/reset mất dữ liệu khi deploy lại - Ephemeral storage). Đồng thời giúp tối ưu hóa dung lượng lưu trữ trên server ứng dụng và tận dụng CDN của Cloudinary để tải ảnh nhanh hơn.

Trả lời: Expo giúp tăng tốc độ phát triển dự án di động nhờ cung cấp sẵn các bộ API phần cứng được tối ưu (như camera, định vị, thông báo push). Nó giúp loại bỏ sự phức tạp khi cấu hình mã nguồn gốc Android/iOS (Java/Swift) ở giai đoạn đầu, cho phép chạy thử ngay trên điện thoại thật thông qua ứng dụng Expo Go mà không cần cài đặt Android Studio hay Xcode nặng nề.

Dự án sử dụng cơ chế Expo Router (file-based routing) tương tự Next.js, giúp quản lý điều hướng giữa 33 màn hình (màn hình login, hồ sơ pet, đặt lịch khám...) một cách trực quan bằng cấu trúc thư mục.

Trả lời: Nhờ kiến trúc Microservices, em có thể tiến hành Scale ngang (Horizontal Scaling) riêng dịch vụ đặt lịch:

  • Tăng số lượng bản sao (replicas) của container appointment-service trong file Docker Compose hoặc Kubernetes.
  • Sử dụng một Load Balancer (ví dụ như Nginx hoặc chính tính năng Load Balancing của Kubernetes/Docker Swarm) để phân phối đều request từ API Gateway đến các bản sao này.
  • Các dịch vụ con khác ít tải hơn (như nutrition-service) vẫn được giữ nguyên để tiết kiệm tài nguyên.

Trả lời: Hiện tại token access_token có thời hạn ngắn (ví dụ: 1 giờ) để đảm bảo an toàn. Khi token hết hạn, API Gateway trả về mã lỗi 401 Unauthorized.

Để tối ưu trải nghiệm người dùng không phải đăng nhập lại liên tục, hướng cải tiến là triển khai cơ chế Refresh Token. Khi đăng nhập, client nhận về cả access_token (hạn ngắn) và refresh_token (hạn dài, lưu dưới DB). Khi access_token hết hạn, interceptor phía client sẽ tự động gọi API gia hạn lên gateway để lấy access_token mới bằng refresh_token một cách âm thầm.

Trả lời: SQL là cơ sở dữ liệu quan hệ (chặt chẽ, hỗ trợ ACID transaction tốt). NoSQL là phi quan hệ (linh hoạt schema, tốc độ ghi nhanh).

Dự án chọn PostgreSQL (một hệ quản trị CSDL quan hệ mạnh mẽ) vì các dữ liệu như Lịch hẹn khám (Appointments), Hồ sơ bệnh án (Medical Records), và Lịch tiêm phòng (Vaccinations) đòi hỏi tính nhất quán, ràng buộc quan hệ cao, giao dịch chặt chẽ để tránh lỗi dữ liệu lịch khám trùng lặp.

Trả lời: Trong mô hình Database-per-service, việc xóa một bản ghi có liên quan ở database khác (ví dụ: Xóa User ở pet_care_auth nhưng còn thú cưng ở pet_care_pets) rất nhạy cảm:

  • Giải pháp hiện tại: Sử dụng phương pháp Soft Delete (Xóa mềm) hoặc Vô hiệu hóa tài khoản (Set trạng thái active = false) thay vì xóa vật lý khỏi database. Cách này giữ nguyên tính toàn vẹn dữ liệu lịch sử lịch hẹn/tiêm chủng đã xảy ra.
  • Giải pháp nâng cao: Gửi một Event bất đồng bộ qua Message Broker thông báo cho Pet-service tự động xóa tất cả thú cưng của người dùng đó sau khi tài khoản bị xóa (Cascade Delete ở tầng ứng dụng).

Trả lời: Dự án tuân thủ cấu trúc chuẩn của NestJS:

  • Mỗi thực thể nghiệp vụ là một Module riêng biệt (chứa Controller, Service, Entity, DTO).
  • Controller: Đảm nhận nhiệm vụ định nghĩa các router endpoint, tiếp nhận đầu vào request và trả kết quả cho client.
  • Service: Chứa toàn bộ logic nghiệp vụ tính toán, xử lý dữ liệu và tương tác trực tiếp với Database.
  • Entity: Khai báo ánh xạ cấu trúc bảng dữ liệu xuống DB qua TypeORM.
  • DTO (Data Transfer Object): Kiểm soát định dạng và kiểm thử tính hợp lệ của dữ liệu gửi lên (sử dụng class-validator).
Trong môi trường thực tế, HA được đảm bảo nhờ deploy đa bản ghi (multi-instance) cho từng container dịch vụ kết hợp Load Balancer. Vì các dịch vụ con ở trạng thái Stateless (không lưu trạng thái phiên đăng nhập trực tiếp trên RAM máy chủ, mọi thông tin lấy từ token JWT và database), nên ta có thể tùy ý khởi tạo hàng chục máy chủ chạy cùng một dịch vụ con để phân tải mà không sợ mất đồng bộ phiên làm việc.
Tuyệt đối không! Chuỗi JWT chỉ được mã hóa dạng Base64 chứ không được mã hóa bảo mật (hacker có thể dễ dàng giải mã chuỗi Payload của JWT để xem thông tin bên trong). Do đó, payload của JWT trong dự án chỉ lưu các thông tin công khai cơ bản: sub (User ID), email, và role để xác thực quyền hạn. Không chứa mật khẩu hay thông tin cá nhân bảo mật nào khác.
API Gateway không kết nối trực tiếp đến các PostgreSQL database của các dịch vụ. Nó chỉ kết nối tới các container dịch vụ con qua HTTP. Nếu bản thân các dịch vụ con mất kết nối với database của chúng, TypeORM sẽ kích hoạt cơ chế tự động kết nối lại (Retry). Nếu thất bại, dịch vụ con sẽ trả về lỗi 500 Internal Server Error và API Gateway sẽ chuyển đổi mã lỗi này gửi về client một cách rõ ràng.
DTO (Data Transfer Object) định nghĩa định dạng dữ liệu truyền tải giữa client và server cho mỗi request. DDTO kết hợp với thư viện class-validator (ví dụ `@IsEmail()`, `@IsString()`, `@IsNotEmpty()`) hoạt động như một bộ lọc giúp tự động kiểm tra định dạng dữ liệu đầu vào. Nếu dữ liệu client gửi lên sai kiểu hoặc thiếu trường bắt buộc, NestJS sẽ tự động chặn và trả lỗi 400 Bad Request ngay từ tầng Controller, bảo vệ tầng Service khỏi dữ liệu rác.

TypeORM là một thư viện ORM (Object-Relational Mapping) giúp ánh xạ các bảng database thành các Class đối tượng TypeScript thông thường. Lợi ích:

  • Bảo mật: Tự động phòng chống tấn công SQL Injection bằng cách chuẩn hóa các tham số đầu vào (Parameterized Queries).
  • Code sạch: Thực hiện các thao tác CRUD thông qua các phương thức OOP trực quan như find(), save(), delete() mà không cần viết các chuỗi lệnh SQL dài dòng, dễ sai sót chính tả.
  • Dễ thay đổi DB: Giúp dự án dễ dàng chuyển đổi hệ quản trị CSDL trong tương lai (ví dụ từ Postgres sang MySQL) mà gần như không phải sửa lại code logic vì ORM tự động dịch sang cú pháp DB tương ứng.
Khi khách hàng đặt lịch hẹn tại appointment-service, hệ thống kiểm tra sự tồn tại của lịch khám trùng lặp đối với cùng một bác sĩ thú y (vet_id) trong cùng một khung giờ dự kiến. Nếu bác sĩ đã có lịch hẹn ở trạng thái CONFIRMED vào giờ đó, hệ thống sẽ trả lỗi và yêu cầu người dùng chọn khung giờ khác.
Thư viện này hoạt động bằng cách nhận request tại API Gateway, phân tích tiền tố của đường dẫn (path prefix), và thiết lập một kết nối socket trung gian chuyển tiếp toàn bộ payload (Headers, Body, Query) của request đó xuống địa chỉ IP nội bộ của container đích (ví dụ: http://pet-service:3003). Nó cũng lắng nghe phản hồi từ container đích để stream ngược kết quả lại cho Client ban đầu, giúp Client cảm giác như đang tương tác với một hệ thống đơn khối duy nhất.
Trong hệ thống, các role được định nghĩa trong Enum gồm: ADMIN, VET, USER. Tại các controller của dịch vụ con, ví dụ PetsController, API lấy tất cả thú cưng của hệ thống /all-pets được gắn tag @Roles(Role.ADMIN, Role.VET). Khi request đi qua RolesGuard, guard này sẽ bóc tách giá trị x-user-role từ header và đối chiếu với danh sách role được phép. Nếu không khớp, sẽ ném ra ngoại lệ ForbiddenException ngay lập tức.
Danh mục vaccine ban đầu được định nghĩa sẵn trong file cấu trúc JSON tĩnh vaccine.json lưu kèm thư mục dự án. Khi chạy container ở bất cứ máy tính hay máy chủ nào, lệnh bootstrap tự động đọc và insert vào database PostgreSQL thông qua TypeORM nếu đếm thấy bảng dữ liệu đang trống. Điều này giúp đảm bảo dữ liệu chạy thử luôn đồng bộ và nhất quán mà không cần chạy file import SQL thủ công.

Điều này giúp quá trình khởi động máy chủ thử nghiệm (Cold Start) và cập nhật nóng mã nguồn (HMR - Hot Module Replacement) diễn ra gần như tức thì, cải thiện đáng kể năng suất lập trình so với các bundler truyền thống như Webpack.
Docker Compose hỗ trợ thuộc tính depends_on. Trong dự án của em:

  • Các dịch vụ con (Pet, Vaccine, Appointment,...) được thiết lập phụ thuộc vào container database postgresdb và redis. Điều này ngăn ngừa tình trạng dịch vụ khởi chạy lên và crash lập tức vì không thể kết nối tới DB đang trong quá trình setup.
